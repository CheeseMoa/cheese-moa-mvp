## 확정 MVP 모델 (코드로 추론 불가 — 반드시 준수)

- 정보구조: 모임(Group) > 이벤트(Event) > 앨범(Album). 앨범 `type`: `person`/`common`/`uncertain`/`eyes_closed`/`blurry`.
- 제작자 인증: **소셜 로그인 4종(카카오·구글·네이버·애플)으로 완전 대체 확정**(CHMO-359, 2026-07-20 — 미배포·사용자 0명이라 마이그레이션·계정 연결 없음). 닉네임+4자리 PIN은 BE 엔드포인트 제거 전까지만 병행. **권한 등급 없음** — 로그인한 모임 멤버는 전원 동일(업로드·검수·공개 가능).
- 모임 진입점 2개(분리): ① **선생님 초대** = 제작자 합류(모임 단위, 모임 비밀번호+참여 링크) ② **학부모 공개** = 무로그인 뷰어(**모임 단위** 공유 링크 + **학부모 전용 비밀번호**(모임 비밀번호와 별개, 모임 생성 시 자동), 조회·다운로드만). 비번 입력 후 **공개된 이벤트 목록**에서 선택.
- 공개(publish) 단위: **이벤트별**(검수 → 공개 요약(14) → 공개하기). 학부모 접근(공유) 단위: **모임별**(공개된 이벤트만 목록 노출). 상세 [[viewer-share-group-level]].
- 검수: **사진 단위 `reviewed`**(앨범 [검토 완료] = 일괄 처리, 앨범 자체엔 검토 상태 없음 — 실 BE는 CHMO-395로 매핑(AlbumPhoto) 단위, 목은 사진 단위 근사) · **재공개 게이트(CHMO-324, 2026-07-20 배포)**: 검토는 표시일 뿐 노출이 아니다 — 뷰어 노출 = 사진 `reviewed && published`(서버 필터링), 발행(`published`)은 [공개하기]가 그 시점 **전 사진 검토 완료된 인물·공통 앨범**의 사진을 내보내는 액션(재호출 유효, 검수하다 만 앨범은 검토분까지 통째로 대기. 발행 대기 수 = 상세 전용 `pendingPublishCount` — published+대기>0이면 14 [공개하기] 재활성·08 안내, FE는 CHMO-265. 05 카드 배지는 목록 응답에 필드가 없어 BE 후속). 검토 해제는 발행을 되돌리지 않고 그 앨범 노출만 숨긴다(CHMO-395) · **공개 후에도 사진 추가·재분석·이동 가능** — `published` 상태 유지, 재분석은 증분(미분류 사진만).
- 타깃: 기관(B2B/유치원) 우선. 일반(B2C) 전용 화면은 후순위.

## 하드 제약

- **BE/AI 코드는 건드리지 않는다.** 이번 범위는 프론트엔드뿐.
- **API 계약 기준은 실 BE다**(오리진 `https://api.cheese-moa.com` — Cloudflare 경유 HTTPS(CHMO-260), EC2 직통 IP는 보안그룹으로 차단, 스웨거 `/v3/api-docs`) — `docs/api-spec.md`와 안 맞는 부분은 **FE를 BE에 맞춘다**. BE 규약: 성공/실패 모두 `{isSuccess, code, message, result}` 봉투 · 숫자 ID(int64) · 경로에 `/api/v1` 프리픽스 없음(vite 프록시가 rewrite) · 시각이 오프셋 없는 UTC(CHMO-205 전까지 FE 보정). 봉투·에러 코드·시각 차이는 `src/api/client.ts`·`src/api/errors.ts`가, 리소스 필드명(`groupId`≠`id`·`eventDate`·`thumbnailUrl`·`personName`)·대문자 enum·서버가 주지 않는 파생값(특수 앨범 표시명·`visibleToViewer`·원본 `url`)은 `src/api/mappers.ts`+도메인 모듈이 흡수한다(CHMO-192) — 화면 코드는 봉투도 BE 필드명도 모른다. 매퍼의 절반은 단순 개명이라 **CHMO-201(BE DTO 필드명 정렬) 확정 후 걷어낼 수 있다** — 그 전엔 움직이는 과녁이라 유지한다. **리프레시 토큰(CHMO-193) 완료** — accessToken(만료 1시간) 401 시 `src/api/client.ts`가 refreshToken으로 1회 자동 재발급(회전 저장·in-flight promise 공유) 후 원 요청 재시도, 실패(TOKEN401) 시 두 토큰 삭제·로그인 복귀. 로그아웃은 `POST /auth/logout`(auth 미첨부, body `{refreshToken}`)으로 서버 무효화 후 로컬 삭제(`src/lib/auth.ts`가 두 토큰 저장 소유). **계약 테스트(CHMO-219) 완료** — 실 BE 응답 픽스처로 봉투·시각 보정·401 재발급·매퍼를 고정했다(`npm run test`). **에러 코드는 실서버에서 채집된 것만 목이 흉내 낸다**(COMMON401·AUTH400/401·TOKEN401·JOIN403·SPACE404·MOMENT404·ALBUM404·PHOTO400/404·VALID400·PUBLISH409 — SPACE404 = 모임 404, 2026-07-16 채집, CHMO-285 · PUBLISH409 = 미검토 공개, 2026-07-22 채집, CHMO-265. PUBLISH400은 BE 소스 대조만). 미채집 케이스(닉네임 중복·이미 멤버)는 FE 의미 코드를 그대로 두고 `// BE 코드 미확인` 주석을 단다 — `errors.ts`가 미지 코드를 통과시키므로 화면은 동작하고, 추측한 코드를 BE 진실 테이블에 굳히지 않는다. 남은 몫: CHMO-196(문서 갱신 — 목이 구현한 계약이 곧 갱신본).
- **BE 결함 4건은 2026-07-10 전부 완료 — 이 제약은 해제됐다**(CHMO-194 착수 중 발견해 티켓을 냈던 것들): 사진 추가 시 기존 사진·앨범·검수 상태 삭제(CHMO-215 — 증분 재분석으로 전환) · 공개 이벤트에 사진 추가 시 `published` 해제(CHMO-216) · 요청당 50장 상한이라 200~300장 경로 없음(CHMO-217) · 분석 실패가 `EMPTY`로 회귀해 미시작과 구분 불가(CHMO-218). 이제 **사진 추가·재분석·분할 업로드를 정상 경로로 다룬다** — 우회 코드를 넣지 않던 이유가 사라졌다. 단 주의 두 가지: ① Jira '완료'일 뿐 **실 BE 배포·동작은 FE가 검증하지 않았다**, 이 경로를 건드리는 스토리는 실 BE로 먼저 확인한다. ② FE의 `MAX_UPLOAD_BATCH = 50`(`src/lib/upload.ts`)은 그대로라 **한 번에 50장이 여전히 FE 상한**이다 — CHMO-217이 BE 상한을 올렸다면 이 값을 맞춰야 하고, 그 전까진 50장씩 나눠 올리면 된다(215가 고쳐져 이제 안전하다).
- FE 개발 기본은 여전히 **MSW 목**(`VITE_ENABLE_MSW=true`) — 실 BE 확인 시 `false`로 끄면 프록시 경유. 실 BE 테스트 계정: `FE연동테스트`/PIN `0709`.
- MVP 제외: 결제/요금제, 다운로드 한도, 휴지통, 인물 병합, 학부모 로그인.

- `npm run typecheck` · `npm run lint` — 검사 · `npm run test`(=`vitest run`)/`test:watch` — **api 계층 계약 테스트**(CHMO-219).
- Prettier: `npm run format`(쓰기)/`npm run format:check`(검사). **주의**: 리포에 기존 CRLF/포맷 잔재가 있어 `format:check`는 클린 체크아웃에서도 실패한다 — 통과/실패 신호로 쓰지 말 것. `format` 전체 실행도 무관 파일까지 재포맷하므로 커밋 전엔 스토리 파일만 스테이징한다.

**워크플로우 스킬**(`.claude/skills/`):
- `/start-story CHMO-###` — 스토리 시작: SP 추천·설정, 상태 '진행 중', `feature/CHMO-###-<slug>` 브랜치 생성.
- `/test-guide` — 현재 브랜치 변경분 기준 테스트 가이드 생성: 자동 검증(typecheck·lint·build) 실행 + "진입 경로 → 조작 → 기대 결과" 수동 체크리스트.
- `/finish-story` — 커밋·푸시·PR 생성(테스트 가이드 포함) + Jira PR 코멘트(best-effort). 머지·브랜치 정리·Jira '완료'는 사용자가 수동으로(스킬은 체크리스트만 안내).
- `/bug-briefing` — 버그·코드리뷰 결함을 "무슨 코드 → 언제 터지나 → 왜 그렇게 됐나 → 어떻게 고치나" 4단 형식 + 코드 위치 링크 + 요약 표로 브리핑.

**가드 훅**(`.claude/settings.json` → `.claude/hooks/`): `main` 직접 commit/push 차단, commit/push 전 typecheck·lint·test(스크립트 정의 시) 실행.
