# bug-briefing — 버그를 사람 말로 브리핑

버그 하나하나를 **개발 배경지식이 얕아도 따라갈 수 있게** 풀어 설명하는 표준 형식.
코드 라인을 다 읽지 않아도 "무엇이 왜 무너지고, 어디를 어떻게 고치는지"가 손에 잡히게 한다.

/ 언제 쓰나
- 코드리뷰(`/code-review`)에서 확정된 결함을 사용자에게 설명할 때
- 방금 고친 버그가 무엇이었는지 브리핑할 때
- 사용자가 "이 버그 뭐야", "왜 터지는 거야", "고친 거 설명해줘"라고 물을 때

## 형식 규칙 (반드시 지킨다)

### 1) 버그마다 4단 구조 — 소제목을 이 순서·이 이름 그대로 쓴다

```
## 버그 N — <한 줄 제목: 무엇이 잘못됐는지> (<file:line 링크>)

**지금 코드가 하는 일**
<이 코드가 원래 뭘 하려는 코드인지 1~3문장. 필요하면 짧은 코드 인용(핵심 몇 줄만,
문제 지점에 `// ①`, `// ②` 주석으로 표시).>

**언제 터지나**
<어떤 입력·상태·순서·타이밍이면 잘못되는지 구체적 시나리오. "예: {name:'x', reviewed:'yes'}"처럼
실제 값으로. 이어서 그래서 사용자/데이터에 무슨 일이 생기는지("왜 심각하냐면 …").>

**왜 그런 구조가 됐나**
<이 코드가 왜 이렇게 짜였는지 — 놓친 불변식, 편의 우선 결정, 호출자 누락 등.
비난이 아니라 원인 규명. 한두 문장이면 충분.>

**어떻게 고치나**
<원칙 한마디(예: "검증 끝난 뒤에만 변이한다") + 고친(또는 고칠) 코드 몇 줄.
이미 고쳤으면 실제 반영 위치를 file:line 링크로.>
```

### 2) 코드 위치는 항상 클릭 가능한 링크로

- 제목과 "어떻게 고치나"에 **`file_path:line_number` 형식**을 쓴다(터미널에서 클릭됨).
  예: `src/mocks/handlers/albums.ts:96`.
- 마크다운 링크로 걸 때: `[albums.ts:96](src/mocks/handlers/albums.ts#L96)`.
- **이미 수정한 버그면** "어떻게 고치나" 끝에 실제 고친 위치를 링크로 명시
  (`→ 반영: src/mocks/handlers/events.ts:164`). 리뷰 단계라 아직 안 고쳤으면 제안 코드만.
- 링크할 라인은 추측하지 말 것 — Grep/Read로 현재 실제 라인을 확인해 정확히 건다.

### 3) 마지막에 한눈 요약 표

전체 버그를 아래 3열 표로 닫는다(라인은 인지 부담이라 넣지 않는다):

```
| | 원인 한 줄 | 수정 규모 |
|---|---|---|
| 1. <짧은 이름> | <왜 터졌나 한 줄> | <예: 1줄 추가 / 순서 재배치 ~15줄 / ~6줄 교체> |
```

표 아래에 버그 간 관계 한 줄(예: "셋 다 독립적이고 계약(api-spec) 변경은 없습니다").

## 톤·분량

- **한국어**, 존댓말. 청중은 "코드를 라인 단위로 읽기는 버거운 발주자/기획자".
- 전문용어는 쓰되 처음 나올 때 괄호로 풀어 준다(예: "이름전파(같은 personId 앨범이 함께 바뀜)").
- 버그당 브리핑은 **화면 한 눈에 들어오는 분량**(코드 인용 포함 15줄 안팎). 장황하면 실패.
- 심각도 순으로 정렬(정합성 버그 먼저, 잠재/저심각 뒤). 필요하면 🔴 확정 / 🟡 잠재로 표시.
- 과장 금지: "엄청난 버그"가 아니면 규모를 솔직히("명세대로 한 곳이 덜 반영된 정도").

## 절차

1. **입력 확보** — 브리핑 대상 결함 목록을 모은다(리뷰 결과, 방금 수정한 diff, 또는 사용자가 가리킨 버그).
2. **라인 확인** — 각 결함의 현재 정확한 `file:line`을 Read/Grep으로 확인한다(링크가 어긋나면 신뢰가 깨진다).
3. **4단 작성** — 위 형식으로 버그마다 작성. 코드 인용은 문제 지점 몇 줄만.
4. **요약 표** — 3열 표로 닫고, 버그 간 관계 한 줄 첨부.
5. **다음 행동 제안**(선택) — 미수정이면 "승인하시면 이 순서로 고치겠습니다", 수정 완료면 검증 결과 한 줄.

## 참고
- 이 저장소는 소스 오브 트루스가 `docs/`(feature-spec·api-spec·screen-spec)다. "왜 이게 버그냐"는
  근거를 코드가 아니라 **명세의 규칙**에서 끌어오면 설득력이 높다(예: "feature-spec의 'ready = 모든 사진
  검토완료' 불변식 위반").
- 관련: 테스트 방법은 [[test-guide]], 리뷰 자체는 `/code-review`.
