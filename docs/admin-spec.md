## 1. 목적

운영 중 나오는 질문에 DB 직접 조회 없이 답하기 위한 백오피스.

- "가입자·모임이 얼마나 늘고 있나"
- "어느 유치원이 실제로 쓰고 있나"
- "이 모임 상태가 지금 어떤가" (문의 대응)

**범위 밖**: 기관 관리자용 모임 내 관리, 결제/요금제, 고객 문의 시스템.

## 2. 확정 결정 3가지

### 2-1. 배포 위치 — 같은 리포 `/admin` 라우트

**판단 근거**

- 모바일 프레임(`PhoneShell`)은 라우터가 아니라 **페이지마다 개별로 두른다**(`src/router.tsx`, 21개 페이지가 각자 import). `/admin` 페이지는 안 두르면 그만이라 "모바일 셸과 섞인다"는 문제가 실제로는 없다.
- 재사용되는 것: `src/api/client.ts`(응답 봉투 언랩 + **401 리프레시 토큰 회전·in-flight 공유**), `src/api/errors.ts`, `src/lib/auth.ts`, `useApi`/`useMutation`, 타입 계약. 특히 리프레시 로직은 미묘해 다시 짜면 버그가 나기 쉽다.
- 배포·CI가 1개로 유지된다(Vercel 프로젝트 1개, `vercel.json`이 `/api/v1/*`를 실 BE로 rewrite).

**대가와 대응**

| 대가 | 대응 |
|---|---|
| 번들 증가 | 라우트를 `lazy()`로 분리(`/dev/components` 선례) — `/admin` 진입자만 받는다 |
| 학부모 뷰어와 같은 도메인 | 1차는 조회 전용이라 수용. 쓰기 액션이 들어올 때 재검토 |

### 2-2. 나중에 별도 리포로 격리하기 위한 규칙 3개 (지금부터 지킨다)

이 규칙을 지키면 격리는 "폴더 옮기고 Vercel 프로젝트 하나 파기"로 끝난다. 안 지키면 실 뽑듯 뜯어내야 한다.

1. **관리자 코드는 `src/admin/` 한 폴더에만.** 서비스 코드가 관리자 코드를 import하지 않는다(의존 방향 한쪽).
2. **공유는 바닥 인프라뿐** — `api/client.ts`·`lib/auth.ts` 등. 화면 컴포넌트는 서로 쓰지 않는다.
3. **BE 경로를 처음부터 `/admin/*`으로 분리.** 프론트를 떼어내도 BE는 그대로 둔다.

### 2-3. 인증 — 유저에 관리자 역할(role) 부여

**핵심 원칙**

> **기존 서비스 API는 role을 쳐다보지 않는다.** 관리자여도 자기가 멤버가 아닌 모임은 일반 화면에서 볼 수 없다. 전체 데이터는 `/admin/*` 전용 API에서만 보인다.

- role은 **더하는 것이지 바꾸는 것이 아니다** — 관리자 계정도 평범한 유저로서 모임 생성·업로드·검수·공개를 그대로 한다(테스트 계정으로 겸용 가능).
- 초기 관리자 지정은 DB 직접 변경으로 충분하다.
- 기본값은 일반 유저 — 기존 행은 마이그레이션 영향이 없다.
- 소셜 로그인 전환(CHMO-359·363) 중이라 **인증 체계를 하나 더 만들지 않는다**. 별도 관리자 계정 체계는 규모가 커진 뒤 재검토하며, 그때 바꾸는 것은 `/admin/*` 앞의 인가 검사뿐이고 서비스 API는 손대지 않는다.

## 3. 화면

### 3-1. 대시보드 (첫 화면)

**지표 카드**

| 항목 | 비고 |
|---|---|
| 총 가입자 수 | |
| 총 모임 수 | |
| 총 이벤트 수 | |
| 총 사진 수 | |
| 최근 7일 신규 모임 수 | `createdAt` 기준 |
| 최근 7일 신규 이벤트 수 | `createdAt` 기준 |
| 최근 7일 업로드 사진 수 | ✅ 사진 업로드 시각이 DB에 있음(2026-07-21 BE 확인) |

**최근 생성된 모임 5개** — 모임 상세로 바로가기.

**넣지 않는 것**

- **시계열 그래프** — 숫자 카드로 "크고 있나"에는 답이 된다. 시계열은 집계 저장이 따로 필요해 비용이 급증한다. 2차 이후.
- **S3 용량·서버 상태·에러 로그** — Grafana/CloudWatch가 이미 본다. 경계: **인프라 지표는 Grafana, 서비스 지표는 어드민.**

### 3-2. 모임 목록

| 컬럼 | 비고 |
|---|---|
| 모임명 | 상세로 링크 |
| 멤버 수 | |
| 이벤트 수 | |
| 사진 수 | |
| 생성일 | |

- 검색: 모임명
- 정렬: 최신순(기본) / 사진 많은 순
- **페이지네이션 필수** — 전체 조회는 데이터가 늘면 그대로 장애가 된다

### 3-3. 모임 상세

- **헤더**: 모임명, 생성일, 멤버 수
- **멤버 목록**: 닉네임, 합류일
- **이벤트 목록**: 이름, 날짜, 상태(빈/분석중/검수중/공개), 사진 수, 앨범 수, 공개일

**학부모 공유 비밀번호는 1차에서 노출하지 않는다** — 조회 전용 화면에 평문 비밀번호가 떠 있으면 화면 캡처 하나가 사고가 된다. 운영 문의는 선생님 본인 화면(05 학부모 공유 시트)으로 안내한다.

## 4. 필요한 API (초안 — 확정은 CHMO-378)

전부 `/admin/*` 아래. 기존 응답 봉투(`{isSuccess, code, message, result}`)·숫자 ID 규약을 따른다.
경로 명칭은 BE 도메인 네이밍(space/moment)을 따를지 BE가 결정한다.

| 용도 | 초안 | 응답 |
|---|---|---|
| 대시보드 요약 | `GET /admin/stats` | 총계 4종 + 최근 7일 3종 + 최근 모임 5개 |
| 모임 목록 | `GET /admin/spaces?page&size&q&sort` | 모임 + 집계 컬럼, 페이지 정보 |
| 모임 상세 | `GET /admin/spaces/:id` | 모임 + 멤버 목록 + 이벤트 목록 |

**설계 주의**

- 목록은 전부 페이지네이션 전제.
- 대시보드 집계를 매 요청마다 `count`로 계산하면 데이터가 늘수록 느려진다. 현재 데이터량에서 측정해보고 캐시·집계 테이블 필요 여부를 판단한다(지금 규모면 불필요할 가능성이 높다).
- 응답에 학부모 공유 비밀번호를 담지 않는다(§3-3).
- 비관리자 토큰으로는 전 엔드포인트가 403.

## 5. BE에 확인해야 할 것

기획 시점에 FE 계약(`src/types/api.ts`)만으로는 확정할 수 없는 항목. 답에 따라 §3 항목이 빠질 수 있다.

| # | 확인 사항 | 상태 | 영향 |
|---|---|---|---|
| 1 | 사진에 **업로드 시각**이 저장되나 | ✅ 있음 (2026-07-21) | "최근 7일 업로드 사진 수" 1차 포함 확정 |
| 2 | 사진에 **파일 크기**가 저장되나 | 미확인 | 용량 지표(2차)의 전제 |
| 3 | **최근 접속 기록**을 남기나 | 미확인 | 유저 활동성 지표(2차)의 전제 |
| 4 | **분석 실패**를 구분해 저장하나 | 미확인 | CHMO-218 이전엔 실패가 '빈 이벤트'로 회귀해 미시작과 구분 불가였다. 고쳐졌는지에 따라 "실패한 분석" 목록(2차) 가능 여부가 갈린다 |

## 6. 2차 이후 (지금 만들지 않음)

- **운영 액션(쓰기)** — 모임·이벤트 강제 삭제, 분석 재시도, 공개 해제, 유저 비활성화. 쓰기가 들어오면 **감사 로그(누가·언제·무엇을)가 필수**가 된다.
- **B2B 기관 등록 검수** — 등록 신청을 사람이 눈으로 확인해 승인/거절. 어드민이 장기 자산이 되는 지점이라, 이때 위치·권한 설계를 다시 본다.
- **유저 목록·상세** — 소셜 로그인 전환으로 유저 스키마가 바뀌는 중이라 미룬다.
- **시계열 그래프**, **분석 job 모니터**.
